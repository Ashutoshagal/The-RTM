         <h3 class="h4 tPadding0 bPadding0 summary">
            <span class="expand_section">XQuery 3.1, XQueryX 3.1, XPath 3.1 and supporting documents now a W3C Recommendation</span>
         </h3>
         <p class="date">
            <span class="dtstart published" title="2017-03-22T14:25:58Z"> 22 March 2017 </span> | <a title="Archive: XQuery 3.1, XQueryX 3.1, XPath 3.1 and supporting documents now a W3C Recommendation" href="https://www.w3.org/blog/news/archives/6197?pk_campaign=feed&amp;pk_kwd=xquery-3-1-xqueryx-3-1-xpath-3-1-and-supporting-documents-now-a-w3c-recommendation">Archive</a>
         </p>
      </div>
      <div class="description expand_description"><p><a class="imageLink" href="https://www.w3.org/blog/news/files/2017/03/Screen-Shot-2017-03-22-at-15.20.14-.png"><img src="https://www.w3.org/blog/news/files/2017/03/Screen-Shot-2017-03-22-at-15.20.14--300x177.png" alt="Screenshot of W3C Standards showing XQuery and XPath 3.1 documents " width="200" class="size-medium wp-image-6199" srcset="https://www.w3.org/blog/news/files/2017/03/Screen-Shot-2017-03-22-at-15.20.14--300x177.png 300w, https://www.w3.org/blog/news/files/2017/03/Screen-Shot-2017-03-22-at-15.20.14--768x454.png 768w, https://www.w3.org/blog/news/files/2017/03/Screen-Shot-2017-03-22-at-15.20.14--1024x605.png 1024w, https://www.w3.org/blog/news/files/2017/03/Screen-Shot-2017-03-22-at-15.20.14-.png 1236w" sizes="(max-width: 300px) 100vw, 300px" /></a>The <a href="https://www.w3.org/XML/Query/" target="_blank">XML Query Working Group</a> and the <a href="https://www.w3.org/Style/XSL/" target="_blank">XSLT Working Group</a> have published six documents as W3C Recommendations to strengthen JSON and Web Platform support through maps, arrays, new functions:</p>
<ul>
<li><a href="https://www.w3.org/TR/2017/REC-xquery-31-20170321/" target="_blank">XQuery 3.1: An XML Query Language</a></li>
<li><a href="https://www.w3.org/TR/2017/REC-xqueryx-31-20170321/" target="_blank">XQueryX 3.1</a></li>
<li><a href="https://www.w3.org/TR/2017/REC-xpath-31-20170321/" target="_blank">XML Path Language (XPath) 3.1</a></li>
<li><a href="https://www.w3.org/TR/2017/REC-xpath-datamodel-31-20170321/" target="_blank">XQuery and XPath Data Model 3.1</a></li>
<li><a href="https://www.w3.org/TR/2017/REC-xpath-functions-31-20170321/" target="_blank">XPath and XQuery Functions and Operators 3.1</a></li>
<li><a href="https://www.w3.org/TR/2017/REC-xslt-xquery-serialization-31-20170321/" target="_blank">XSLT and XQuery Serialization 3.1</a></li>
</ul>
<p>The 3.1 work extends XPath and XQuery with map and array data structures along with additional functions and operators for manipulating them; a primary motivation was to enhance JSON support.</p>
<p>XPath is a domain-specific language for identifying and extracting nodes from a tree (typically) built from XML or JSON and defined by the XQuery and XPath Data Model) and also an expression language with typed tree nodes and functions among the first-class objects. XPath expressions can call functions and use operators defined in the Functions and Operators 
specification.</p>
<p>XQuery extends XPath to support complex joins, windowing, grouping and other operations across collections of trees that are potentially stored in high-performance indexed databases.</p>
<p>XQueryX is an XML representation of XQuery expressions.</p>
<p>The Serialization specification describes how results of XPath and XQuery(X) expressions can be delivered in HTML, XHTML, JSON, XML or text.</p>

         </h3>
         <p class="date">
            <span class="dtstart published" title="2017-03-30T16:18:04Z"> 30 March 2017 </span> | <a title="Archive: W3C Invites Implementations of WebDriver" href="https://www.w3.org/blog/news/archives/6206?pk_campaign=feed&amp;pk_kwd=w3c-invites-implementations-of-webdriver">Archive</a>
         </p>
      </div>
      <div class="description expand_description"><p>The <a href="https://www.w3.org/testing/browser">Browser Testing and Tools Working Group</a> invites implementation of the Candidate Recommendation of <a href="https://www.w3.org/TR/2017/CR-webdriver-20170330/">WebDriver</a>. WebDriver is a remote control interface that enables introspection and control of user agents. It provides a platform and language-neutral wire protocol as a way for out-of-process programs to remotely instruct the behavior of web browsers as well as a set of interfaces to discover and manipulate DOM elements in web documents and to control the behavior of a user agent. It is primarily intended to allow web authors to write tests that automate a user agent from a separate controlling process, but may also be used in such a way as to allow in-browser scripts to control a possibly separate browser. </p>

            <span class="expand_section">Spatial Data on the Web Best Practices Note Published</span>
         </h3>
         <p class="date">
            <span class="dtstart published" title="2017-03-30T16:18:04Z"> 30 March 2017 </span> | <a title="Archive: Spatial Data on the Web Best Practices Note Published" href="https://www.w3.org/blog/news/archives/6209?pk_campaign=feed&amp;pk_kwd=spatial-data-on-the-web-best-practices-note-published-2">Archive</a>
         </p>
      </div>
      <div class="description expand_description"><p>The <a href="//www.w3.org/2015/spatial/">Spatial Data on the Web Working Group</a> has published a Group Note of <a href="https://www.w3.org/TR/2017/NOTE-sdw-bp-20170330/">Spatial Data on the Web Best Practices</a>. This document advises on best practices related to the publication and usage of spatial data on the Web; the use of Web technologies as they may be applied to location. The best practices are intended for practitioners, including Web developers and geospatial experts, and are compiled based on evidence of real-world application. These best practices suggest a significant change of emphasis from traditional Spatial Data Infrastructures by adopting a Linked Data approach. As location is often the common factor across multiple datasets, spatial data is an especially useful addition to the Linked Data cloud; the 5 Stars of Linked Data paradigm is promoted where relevant.</p>

            <span class="expand_section">W3C Invites Implementations of Resource Timing Level 1</span>
         </h3>
         <p class="date">
            <span class="dtstart published" title="2017-03-30T16:18:03Z"> 30 March 2017 </span> | <a title="Archive: W3C Invites Implementations of Resource Timing Level 1" href="https://www.w3.org/blog/news/archives/6211?pk_campaign=feed&amp;pk_kwd=w3c-invites-implementations-of-resource-timing-level-1-2">Archive</a>
         </p>
      </div>
      <div class="description expand_description"><p>The <a href="http://www.w3.org/webperf">Web Performance Working Group</a> invites implementation of the Candidate Recommendation of <a href="https://www.w3.org/TR/2017/CR-resource-timing-1-20170330/">Resource Timing Level 1</a>. This specification defines an interface for web applications to access the complete timing information for resources in a document. User latency is an important quality benchmark for Web Applications. While JavaScript-based mechanisms can provide comprehensive instrumentation for user latency measurements within an application, in many cases, they are unable to provide a complete end-to-end latency picture. This document introduces the PerformanceResourceTiming interface to allow JavaScript mechanisms to collect complete timing information related to resources on a document. Navigation Timing 2 extends this specification to provide additional timing information associated with a navigation. </p>

            <span class="dtstart published" title="2017-03-20T16:39:30Z"> 20 March 2017 </span> | <a title="Archive: Call for Review: Encrypted Media Extensions Proposed Recommendation Published" href="https://www.w3.org/blog/news/archives/6194?pk_campaign=feed&amp;pk_kwd=call-for-review-encrypted-media-extensions-proposed-recommendation-published">Archive</a>
         </p>
      </div>
      <div class="description expand_description"><p>The <a href="https://www.w3.org/html/wg/">HTML Media Extensions Working Group</a> has published a Proposed Recommendation of <a href="https://www.w3.org/TR/2017/PR-encrypted-media-20170316/">Encrypted Media Extensions</a>. The specification extends HTMLMediaElement to define a common API to control playback of protected content. EME does not define a content protection or Digital Rights Management system. The common API may be used to discover, select and interact with such systems as well as with simpler content encryption systems. Comments are welcome through <strong>13 April</strong>. Please, read more in the <a href="https://lists.w3.org/Archives/Public/public-html-media/2017Mar/0016.html">public announcement</a>.</p>
